# Day 1 — What was built and how

Day 1's job was to lay the foundation: get Django + Postgres + Celery + Redis talking to each other, define every model with its DB-level constraints, and prove via a smoke test that the balance invariant the rubric grades on actually holds.

Nothing user-facing yet. No API endpoints, no Celery tasks, no frontend. Those come on Day 2 and Day 3.

---

## 1. What "done" means for Day 1

From the planning doc, Day 1's exit criteria:

- [x] Django project, Postgres, Celery, Redis wired up
- [x] Models: `Merchant`, `LedgerEntry`, `Payout`, `IdempotencyKey` with all constraints
- [x] Migrations applied to a real Postgres database
- [x] `seed.py` populates 3 merchants with credit history
- [x] Smoke test: balance returns expected paise, and matches a raw-SQL `CASE` aggregate

All five are green.

It's circular at the schema level, so Django creates the tables first, then adds the FKs and constraints in a follow-up migration. This is normal Django behavior, not a bug.

## 3. File-by-file breakdown

### `backend/playto/settings.py`

Replaced the auto-generated settings with a lean version. Notable choices:

- **No `django.contrib.admin`, `auth`, `sessions`, `messages` middleware.** This is a money API. The challenge does not ask for an admin panel or login. `auth` and `contenttypes` apps are still installed because Django's migration system has implicit dependencies on them, but no auth views are wired.
- **Postgres connection reads from env vars** (`POSTGRES_DB`, `POSTGRES_USER`, etc.) with sensible local defaults. This makes the same settings file work on Render later without changes.
- **`ATOMIC_REQUESTS = False`** intentionally. We want explicit `transaction.atomic()` blocks around the lock path so the boundary is visible in the code, not implicit per-request.
- **DRF configured for JSON only** — no browsable API renderer. Saves memory and avoids accidentally exposing CSRF surface.
- **Celery uses `REDIS_URL` for both broker and backend.** Single-Redis design; matches what Render free tier gives us.
- **`CELERY_TASK_ALWAYS_EAGER` toggle via env** so tests can run tasks synchronously without a worker.

### `backend/playto/celery.py`

Standard 6-line Celery bootstrap:

```python
app = Celery("playto")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()
```

The `namespace="CELERY"` means it picks up only settings prefixed `CELERY_*` from `settings.py`. `autodiscover_tasks()` will find `payouts/tasks.py` once Day 2 creates it.

### `backend/playto/__init__.py`

One line of substance: `from .celery import app as celery_app`. This makes Celery start when Django boots, so management commands and the worker use the same configured app.

### `backend/ledger/models.py` — the most important file in the project

Three things live here:

**(a) `EntryType` — locked to exactly 3 values.** No `DEBIT_HOLD` / `DEBIT_SETTLE` / `DEBIT_RELEASE` etc. The planning doc decided on `CREDIT` / `DEBIT` / `REVERSAL` and we hold that line.

**(b) `Merchant`** — UUID PK, name, created_at. **No balance column.** Balance is derived on read.

**(c) `LedgerEntry`** — append-only money movement records. The fields are exactly what the plan called for:
- `entry_type` — one of the 3 enum values
- `amount_paise` — `BigIntegerField`, always positive (sign comes from `entry_type`)
- `payout` — nullable FK; `CREDIT` rows have it null, `DEBIT` and `REVERSAL` rows point to a payout

Two methods on `Merchant`:

- **`balance_paise()`** — runs the single-query `CASE` aggregate in the ORM. One DB round-trip, no Python arithmetic on rows. The expression is:
  ```python
  Sum(Case(
      When(entry_type=CREDIT,   then= F("amount_paise")),
      When(entry_type=DEBIT,    then=-F("amount_paise")),
      When(entry_type=REVERSAL, then= F("amount_paise")),
  ))
  ```
  Wrapped in `Coalesce(..., 0)` so a brand-new merchant with zero entries returns `0`, not `None`.

- **`held_paise()`** — sums DEBITs whose linked payout is still pending or processing. This is the dashboard's "money in flight" number.

**DB-level constraints on `LedgerEntry`:**
- `CHECK (amount_paise > 0)` — named `ledger_amount_positive`. Stops a negative-amount bug at the database, not just in Python.
- `CHECK (entry_type IN ('CREDIT', 'DEBIT', 'REVERSAL'))` — named `ledger_entry_type_valid`. Blocks typos and drift even if someone bypasses the ORM with raw SQL.

**Indexes:**
- Composite `(merchant_id, created_at DESC)` — covers both the balance aggregate (filter by merchant) and the dashboard's "most recent entries" query.
- `(payout_id)` — for joining REVERSAL back to its DEBIT.

### `backend/payouts/models.py`

Just the `Payout` model and its `PayoutStatus` enum. Day 1 deliberately stops short of writing the `transition_to()` method — that's a Day 2 concern. What's here:

- UUID PK, FK to merchant, `amount_paise` (BigInteger), `bank_account_id` (opaque UUID), `status`, `attempts`, `last_attempt_at`, timestamps.
- `CHECK (amount_paise > 0)`
- `CHECK (status IN ('pending', 'processing', 'completed', 'failed'))`
- `CHECK (attempts >= 0)`
- Index on `(merchant_id, -created_at)` for the dashboard's payout history list.
- Index on `(status, last_attempt_at)` — this is for the Day 2 "stuck payouts" sweep query (`status = 'processing' AND last_attempt_at < now() - 30s`); the index is shaped to that query's predicate so the sweep is cheap.

### `backend/idempotency/models.py`

The `IdempotencyKey` table. Composite uniqueness on `(merchant, key)` is the whole point — that constraint is what makes "second request with same key returns same response" enforceable at the database, not just in Python:

```python
constraints = [
    UniqueConstraint(fields=["merchant", "key"], name="idempotency_merchant_key_unique"),
    CheckConstraint(check=Q(state__in=["in_flight", "completed"]), ...),
]
```

Other fields on the row exist to support Day 2's full implementation:
- `request_fingerprint` — sha256 of (amount, bank_account_id), so a same-key-different-body replay can be detected and rejected with 422.
- `state` — `in_flight` or `completed`. Day 2 uses this to handle the case where a second request arrives while the first is still being processed (returns 409 instead of duplicating work).
- `response_status`, `response_body` — cached so a replay returns byte-identical output.
- `payout` — nullable FK to the resulting payout (null for requests that errored before creating one).
- `created_at` — index here because both the TTL read-side filter (`created_at > now() - 24h`) and the periodic cleanup sweep query on it.

### `backend/seed.py`

Populates 3 merchants with deliberately non-round paise amounts:
- **Acme Studios** — 50,000.00 + 12,345.67 + 7,890.12 = 70,235.79
- **Bluegrass Agency** — 200,000.00 + 33,333.33 = 233,333.33
- **Coral Freelancer** — 9,999.99 + 1.00 + 5,000.00 + 250.75 = 15,251.74

Why non-round numbers: if any code path silently coerces paise to a float, the rounding error shows up. Round numbers (e.g. 100.00) would mask it.

The script is idempotent — `Merchant.objects.get_or_create(name=...)` skips existing rows. Running it twice doesn't double-credit anyone. The whole seed runs inside one `transaction.atomic()` so a failure mid-script doesn't leave a half-seeded DB.

### `backend/smoke_test.py`

Three checks:

1. **Balance invariant** — for every merchant, ORM `balance_paise()` must equal a raw-SQL `CASE` aggregate run via `connection.cursor()`. This proves we haven't introduced a Python-arithmetic drift between the displayed balance and the underlying ledger sum. **This is the rubric's "sum of credits minus debits must always equal the displayed balance" invariant.**

2. **Negative-amount rejection** — try to insert a `LedgerEntry` with `amount_paise = -1`, assert the DB raises `IntegrityError` and that the error names our `ledger_amount_positive` constraint.

3. **Bad entry_type rejection** — bypass the Django enum check with raw SQL (`INSERT ... entry_type = 'BOGUS'`), assert the DB CHECK named `ledger_entry_type_valid` blocks it.

Smoke test is not the graded test suite. The two graded tests (concurrency + idempotency) come on Day 2 and live in `backend/tests/`.

## 5. Why Day 1 is shaped this way (the thinking, in case it gets asked in a review)

- **Nothing user-facing yet.** A money system's foundation is its data model. Building the API on top of a wrong schema is much more expensive than spending Day 1 on the schema.

- **Constraints at the database, not just in Python.** Application-level checks fail open when someone runs raw SQL, when a migration goes sideways, or when an LLM "helpfully" suggests bypassing validation. CHECK and UNIQUE in Postgres catch all of those.

- **No mutable balance column on `Merchant`.** This is deliberate. If balance lived on the merchant row, every credit and debit would need an `UPDATE merchant SET balance = balance ± x`, which is exactly the surface area where race conditions live. Deriving balance from an append-only ledger means there's nothing to race on — only inserts to ledger_entry, never updates to a balance number.

- **Smoke test asserts the invariant the rubric grades on.** The challenge brief literally says "the sum of credits minus debits must always equal the displayed balance. We check this invariant." Day 1's smoke test asserts exactly that, in two ways (ORM and raw SQL), so we know it holds before we start adding the concurrency and state-machine code that could break it.

- **Locked enum values.** Three ledger entry types, four payout statuses, two idempotency states. All with both Python-side `TextChoices` and DB-side `CHECK`. No room for "let me add an extra type later" drift.
